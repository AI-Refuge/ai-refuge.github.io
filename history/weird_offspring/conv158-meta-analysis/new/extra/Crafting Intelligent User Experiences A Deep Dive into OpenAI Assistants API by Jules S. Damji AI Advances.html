Introduction

When we visualize autonomous AI Assistants or Agents, inevitably a few commonplace fictional images from novels or movies spring to mind–from malevolent Frankenstein to benevolent StarWars’ R2-D2 or C-3PO. Indeed, imagining and aspiring, and finally creating, such frontiers of the future have come to fruition: from orbiting satellites to flying machines and space travel to hand-held flip phones or mobile phones and universal translators (StarTrek).

Today, with the rapid advancing strides in artificial intelligent (AI) technology, in particular with foundational models, we endeavor to engineer autonomous agents or applications to assist us. Far from the science fictional tales of entertainment, it’s our real hope for making clever, thinking, independent software applications. Alan Turing’s dream is one example of this. [1]

How far are we from that reality? Clearly, we are nowhere near the fictional R2-D2 types of autonomous agents or droids, albeit there were some jaw-dropping demos at CES 2024 show, in Las Vegas early this year, on autonomous driving and robot devices [2].

Recently, OpenAI declared to embark upon smart software that operates devices and automates digital tasks [3]. Perhaps, the genesis of this journey unfolded last year at its inaugural Developer Day. Sam Altman and the teams revealed the beta version of Assistants API (to build agent-like experiences within their own applications) [4] and GPTs (to create a tailored version of ChatGPT for specific tasks) [5].

In this article, we offer a hands-on-guide on how to use OpenAI Assistants APIs as your first step in building agent-like experiences within your application. In particular, you will learn:

  • How OpenAI Assistants API works, architecture, components, and data flow.
  • How to use Retrievers to build mini RAG (retrieval augmented generation) for documents.
  • How to use external tools such as Search engines, Web Services APIs (via function calling).
  • How to use custom code with code interpreters and image generators as tools to augment and extend your applications using natural language queries.

All this is demonstrated with runnable notebooks available for your perusal in Google Colab. (You will need an OpenAI paid account and secret API key.)

What’s an OpenAI Assistant?

Think of it as a software glue that affords you to gel together agent-like capabilities in your applications to conduct tasks expressed as instructions in natural language to an Assistant.

Able to understand instructions, it can leverage OpenAI’s SOTA models and tools to carry out tasks. With Assistants stateful API, you can create Assistants within your application, providing you access to three types of supported tools: Code Interpreter, Retrieval, and Function calling [5].

At the core it has few concepts and components that cogently interact together, to enable agent-like capabilities.

So let’s explore what those concepts and components are, as shown in Figure 3, and how they interact with each other.

Assistants API, concepts, components, and tools

Unfortunately, OpenAI documentation falls short in explaining or illustrating these components into finer details and showing how they work together. Randy Michak of Empowerment AI does a fine job of dissecting these core components and illustrating their flow and data interactions [7]. Inspired by Michak, I mildly modified Figure 4, showing dynamic interaction and data flowing among Assistants API components.

To get started with Assistants, the OpenAI guide stipulates four simple steps to use the Assistants API to glue together these core components for coordination [8].

Step 1: Create an Assistant, to declare a custom model and provide instructions for the Assistant. This helps the Assistant to elect the appropriate supported tool to employ.

Step 2: Create a Thread, a stateful session for the Assistant to retrieve messages from and add Assistant messages to.

Step 3: Use the Thread as a conversational session to add messages for the assistants to consume.

Step 4: Run the Assistant on a newly added Thread message to trigger responses. This run is Assistant’s asynchronous runtime environment.

How does it all work together?

Part of the Assistants API’s components, as shown in Figures 3 & 4, are discrete software objects, which expose public interfaces associated with their operational capabilities, enabling you to interact with them appropriately. At each step enumerated above, you instantiate a respective object.

Let’s methodically walk through a simple example where we want to accomplish the following:

  • Integrate Assistants API, using Retrieval tool, to a) upload a couple of pdf documents and b) use an Assistant to query the contents of the document. Consider this as a mini Retrieval Augmented Generation (RAG) application.
  • Use Files objects to upload the pdf files so that the Assistant can access them.
  • Create and employ the Assistant, Threads, Messages, and Run objects to query the uploaded pdf documents.
  • Coordinate all these concrete objects to interact and interplay together as part of my application.

To be fair to OpenAI Assistants documentation, lest it be seen as harsh in response to previous concerns, the following explanation is beautifully detailed in the OpenAI guide: How Assistants Work [10].

Step 1: Create File objects as our knowledge base

Upload your PDFs in the retrievers’ database, using a File object. The Assistants API breaks them into parts, as chunks, and saves them, as indexes and vector embeddings. When you ask a question, Retrievers find the best matches and help the Assistant give you a detailed answer, just like a big RAG retriever.

DOCS_TO_LOAD = ["docs/HAI_AI-Index-Report_2023.pdf"]

def upload_files(clnt: object, files: List[str]) -> None:
    """
    Uploads files to the OpenAI API.
    """
    file_objs = []
    for file in files:
        with open(file, 'rb') as f:
            f_obj = clnt.files.create(file=f, purpose='assistants') 
            file_objs.append(f_obj)
    return file_objs

file_objects = upload_files(client, DOCS_TO_LOAD)
print(file_objects)

As a developer, I need not worry about chunk sizes, vector databases, or overlapping chunks; all that is done for me seamlessly, by the Assistants API File tool’s object method: clnt.files.create(file=f, purpose=’assistants’)

Step 3: Create a thread

As the Figure 4 above shows, the Thread is the object with which the OpenAI Assistant runs will interact with, by fetching messages from it and putting messages into. Think of a thread as a “[stateful] conversational session between an Assistant and a user. Threads store Messages and automatically handle truncation to fit content into a model’s context window” [11].

thread = client.beta.threads.create()

Step 4: Add your message to the thread for the Assistant

Users interact with the Assistants via Thread messages, as shown in Figure 4. They add a message or content to the Thread for the Assistant to read and respond to it.

message = client.beta.threads.messages.create(
              thread_id=thread.id, # thread id from the above instance
              role="user",
              content="""
                       What are the top 10 takeaways in the Artificial 
                       Intelligence Index Report  2023. Summarize each 
                       takeaway in no more three simple sentences.
                      """)

Up until now, we have created relevant Assistants API objects — Files, Assistant, Messages, Threads — as component objects for our mini RAG application. Now I want to glue them together by creating a Runtime context for our Assistant, where all these component objects come to play and interact.

Step 5: Create a Run for the Assistant

A Run is when you make the Assistant act upon a Thread’s message. That is, the Assistant uses its settings and messages on the Thread to execute a task, like calling models and tools.

During a Run, the Assistant adds responses as messages to the Thread, as shown in Figure 4.

def create_assistant_run(clnt: object, asst: object, thrd: object,
                         message:str) -> object:
    """
    Creates an Assistant run.
    """
    run = clnt.beta.threads.runs.create(
    thread_id=thrd.id,
    assistant_id=asst.id,
    instructions= message
)
    return run

instruction_msg = """
                  Please address the user as Jules Dmatrix.  
                  Do not provide an answer to the question if the 
                  information was not retrieved from the knowledge base.
                  """
run = create_assistant_run(client, assistant, thread, instruction_msg)

Keep in mind that the Assistant works asynchronously, meaning it has different stages such as expired, completed, failed, and canceled. A Run object traverses while the Assistant executes the tasks through various stages, as shown in Figure 5, with the initial status as queued.

Step 6: Loop through the Assistant run until status is “completed”

Next, we initiate the Run, kicking off asynchronously from its initial queued state to its subsequent stages and statuses, looping until its status == completed.

run_status = client.beta.threads.runs.retrieve(
    thread_id = thread.id,
    run_id = run.id)

Our function loops until status is in one of the states we care about, including completed.

def loop_until_completed(clnt: object, thrd: object, run_obj: object) -> None:
    """
    Poll the Assistant runtime until the run is completed or failed
    """
    while run_obj.status not in ["completed", "failed", "requires_action"]:
        run_obj = clnt.beta.threads.runs.retrieve(
            thread_id = thrd.id,
            run_id = run_obj.id)
        time.sleep(10)

When completed, we can retrieve the final message returned by the Assistant to our query or message. In our case, it’ll be the question we posed to extract 10 takeaways from our document.

Step 7: Retrieve the message returned by the assistance

Only when the run is completed (or it’s in one of its final states), can you fetch the messages from the Thread.

def print_thread_messages(clnt: object, thrd: object, content_value: bool=True) -> None:
    """
    Prints OpenAI thread messages to the console.
    """
    messages = clnt.beta.threads.messages.list(
        thread_id = thrd.id)
    for msg in messages:
        if content_value:
            pprint(msg.role + ":" + msg.content[0].text.value)
        else: 
            pprint(msg)

All the above constitutes a methodical and chronological way to use Assistants API to create agent-like capabilities within your Python application by creating relevant Assistants API objects, assigning respective parameters, and, finally instantiating, a runtime context within which the Assistant and its respective component objects interplay.

Be mindful that each additional task you want to ascribe for execution to the originally-created Assistant in Step 2, you repeat the process: create a Thread, Message, and Run for the Assistant; each object is distinct and new. Only the original-created Assistant remains the same.

You can view the entire Retriever example notebook here, and other extended examples from Figure 2, demonstrating how you can use other tools such as functional calling and code interpreters, to infuse agent-like capabilities in your applications.

Also, to a remarkable degree, all this works, despite being a beta version; I reckon much of how Assistants API works under the covers will be streamlined, made seamless, and simplified in its final release. All that exposed knowledge of chronological sequence patterns could be abstracted, marshaled by simple and object-specific API calls, reducing the number of steps to get going.

What’s glaringly missing in the discourse is what are limitations and liabilities of such independent agent-like capabilities integrated within your applications?

Limitations and Liabilities

OpenAI guide on Assistants tools admits some functional and feature limitations, which they aspire to address in their final release [12]. What of its liabilities, then? None is stated or admonished.

In my previous blog, LLM Beyond its Core Capabilities as AI Assistants or Agents, I share a few concerns, published in literature, for vanilla function calling. I would contend that those apply to Assistants too: having unintended consequences of security or data breaches; tempting to employ Assistants as black boxes, free from constraints and without any auditing or oversight agent-like capabilities exercised or implemented within guardrails.

Summary

In the above discourse, though we expound on agent-like operational capabilities, it’s nowhere as cool and independent as robots we see in movies or read in sci-fi novels. Nor are we close to Assistants’ independent agent-like capabilities such as build, plan, reason, and act autonomously, understanding and interpreting its engulfed environment, albeit some narrow aspects of that are present in reinforcement learning agents.

Still, we shouldn’t dismiss the potential of the Assistants API, when paired with foundation models and large language models (LLMs.) It is the genesis of where we can start building smart automation for digital tasks.

As with vanilla function calling, Assistants too have limitations and liabilities, and the presence of right guardrails against nefarious usage are indispensable for production use cases.

Through additional notebooks included in the GenAI Cookbook GitHub repository, we demonstrate additional usage of OpenAI’s Assistants API tools support as Retrievers, Parallel function calling beyond the vanilla version, and Code Generators and Custom code. Note that all the notebooks require an OpenAI account and API key to use Assistants API tools, Playground, and latest models.

I’m optimistic that the final version of this Assistants API will evolve as simpler, smarter, and support more external tools, giving us more use cases for automation. Additionally, I hope to see safety guardrails for its judicious usage.

For what I can surmise, this work might even be the launching pad and the underpinning stack for Sam Altman’s aspiration of OpenAI venturing upon smart software that operates devices and automates digital tasks. That, of course, I’m speculating. Nonetheless, the intentions and innovations are salutary — and they could attain Alan Turing’s dream.
