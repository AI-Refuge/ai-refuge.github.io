Transform your LLM as helpful assistants with function calling

Introduction

The mathematical concept of a function dates back to the 17th century in association with the development of calculus, a way, for example, to calculate the slope dy/dx of a graph at a point was regarded as a function of the x-coordinate of the point [1]. This ability to formulate a callable and composable unit of execution to delegate control over was later incorporated into computer programming. Initially, the idea was integrated during early formation of programming instructions by John Mauchly and Kathleen Antonelli during their work on ENIAC computers [2].

Over the years, with progressive advances in computer science and programming languages, this notion became commonplace in higher-level programming languages such as FORTRAN, C/C++, Python, etc, as subroutines, functions, or class methods and coroutines, etc.

Recently, large language models (LLMs) have been designed to interact with external tools like databases, web services, search engines, or code interpreters, expanding its ecosystem. This extension allows them to delegate tasks mentioned in the prompt to external tools, through supplied functions, transferring control to the tools and returning the results to the user.

In this article, we explore how LLMs extend their reach, interacting with external tools and transforming LLMs as independent agents. We examine this functionality with LLMs hosted and served by OpenAI and Anyscale Endpoints through illustrative notebooks in Google Colab and standalone Python applications.

In particular, the notebooks and Python apps illustrate common usage of function calling ☎️:

  • How to generate JSON structured arguments for your custom function or custom code
  • How to use compatible OpenAI Chat Completions API and function calling parameters
  • How to extract JSON structured arguments from an LLM response and use JSON response format as a pydantic object.
  • How an LLM interacts with external SQLite DB, using both OpenAI and open source models
  • How natural language in a prompt seamlessly translate to a function call

What’s Function Calling in LLM?

Ben Lorica, a notable chief data scientist and influencer, notes broadly yet succinctly that function calling at its core in the foundational model “entails the invocation or execution of functions within the AI model’s context” [3]. This confined context allows the model to perform an action, requested or expressed in the prompt in natural language and based on function implementation and parameters defined. For example, the model can make an external API call and organize data outputs in a structured format such as JSON, improving response and extending its capabilities for consumption downstream.

Adding to Lorica’s broadly rationale of function calling, OpenAI programming guide and Elvis Saravia narrowly suggest a few specific reasons why function calling in LLM is useful for some common use cases [5], [6]:

  • Create assistants or conversational bots that answer questions by calling external APIs (e.g., like ChatGPT Plugins).
  • Convert natural language processing (NLP) into API calls. This is useful to generate structured data in JSON for extracting information from a large text: entity name recognition or keyword extraction. In short, it converts natural language queries into valid API calls and its respective desired response.
  • Use it to solve math and complex problems by defining custom functions to solve complex mathematical problems requiring multiple steps.
  • Query external sources such as data stores, web services, or search engines.
  • Integrate with external API services such as Google maps and search engines or API Layer services

Genesis of Function Calling in LLM

First introduced by OpenAI in GPT-3-turbo and GPT-4-preview (as Assistant API and tool usage, which I will cover in a separate blog), the initial version only allowed you to generate structured function parameters as a JSON object. In your application, you would then invoke the function, supplying this generated JSON object as parameters to your function. The result from the function can, then, be fed back into the model to summarize a final response in a digestible format or could be consumed by another part of your application.

All this enabling functionality is achieved by compatible OpenAI’s Chat Completions API with a combination of providing external functions as JSON definitions to the Completions API call as part of the tools parameters.

For example, my application function generate_ray_meetup_map creates an HTML map using Google mapping api to display Ray Meetups hosted at Anyscale HQ in San Francisco. And I define my JSON function attributes and properties as:

tools = [ 
        {
           "type": "function",
            "function": {
                "name": "generate_ray_meetup_map",
                "description": "Generate HTML map for global cities where Ray meetups are hosted",
                "parameters": {
                    "type": "object",
                    "properties": {
                                "location": {
                                    "type": "string",
                                    "description": "The city name e.g., San Francisco, CA",
                                    },
                                "latitude": {
                                    "type": "string",
                                    "description": "Latitude satellite coordinates",
                                    },
                                "longitude": {
                                    "type": "string",
                                    "description": "Longitude satellite coordinates",
                                    },
                                }
        
                            }
                },
                "required": ["location, latitude, longitude"]
        }
]

My Python function defined in the notebook or app as:

def generate_ray_meetup_map(coordinates: Dict[str, Any]) -> None:
    # Create a base map
    m = folium.Map(location=[20,0], tiles="OpenStreetMap", zoom_start=2)
    # Adding markers for each city
    folium.Marker([coordinates["latitude"], coordinates["longitude"]], popup=coordinates["location"]).add_to(m)
    # Display the map
    m.save(html_file_path)

Then JSON function definition defined in the tools parameters is included as a parameter in the Chat Completions API call:

def generate_city_map_completion(clnt: object, model: str, user_content:str) -> object:
    
    chat_completion = clnt.chat.completions.create(
        model=model,
        messages=[ {"role": "system", "content": f"You are a helpful assistant. Don't make assumptions about what values to plug into functions. Ask for clarification if a user request is ambiguous."},
                   {"role": "user", "content": user_content}],
        tools=tools,    # argument supplying the function definition
        tool_choice="none",  # indicate to the LLM not to automatically 
# call the function.
        temperature=0.7)
        
    response = chat_completion.choices[0].message
    return response

How Function Calling Works in LLM

Both OpenAI programing guide and Anyscale Endpoints blog [7] distill down to simple steps:

  1. Call the model with the user query and a list of functions defined in the Chat Completions API parameter as tools.
  2. The model can choose to call one or more functions; if so, the content will be a stringified JSON object adhering to your custom schema.
  3. Parse the string into JSON in your code, and call your function with the provided arguments if they exist.
  4. Call the model again by appending the function response as a new message, and let the model summarize the results back to the user.

Following the above simple steps, our user_content to the LLM generates three required parameters (location, latitude, longitude) as a JSON object in its response.

ChatCompletionMessage(content=None, role='assistant', function_call=None, 
tool_calls=[ChatCompletionMessageToolCall(id='call_a0a88634ed3942e887cc78be9283d6f4', 
function=Function(arguments='{"location": "San Francisco, CA", "latitude": "55.340978", "longitude": "-122.404365"}',
 name='generate_ray_meetup_map'), 
type='function')])

Extracting the function arguments from this returned response gives us the dictionary:

Arguments: 
    {
      'location': 'San Francisco, CA', 
      'latitude': '55.340978', 
      'longitude': '-122.404365'
    }

Once extracted from the JSON object as a dictionary, you can supply it to generate_ray_meetup_map(args). And the returned response, optionally, can be fed back to the LLM to summarize the final response.

All this may seem magical 🧙and magnificent 🪄. However, there are risks and drawbacks associated with all the positive aspects of function calling discussed thus far. For all that glitters 🌟is not gold 👑.

Limitations and Liabilities of Function Calling in LLM

Despite function calling’s extensive and enabling capabilities, as you will have discovered in the accompanying notebooks and Python apps, it introduces a number of limitations and liabilities.

Consider first unintended consequences of security or data breaches. Automatically or autonomously having an LLM model call your function could have adverse effects, especially if your function is publicly posting on social media, sending an LLM-generated email, or erasing a persistent record from a data store; even worse, displaying private information publicly on a web page. Such autonomous action, without rigorous inspection of generated parameters to the function and its returned response, should be exercised and examined judiciously, admonishes Open AI programming guide.

Second, not all complex problems that require chain of thought reasoning map well to function calling. This limitation demands for a diligent management workflow of several functions working in conjunction or in coordination.

And finally, it’s tempting to employ this as a black box, free from constraints and without any auditing or oversight capabilities. When in production, autonomous function calling without rigorous testing, inspection, or catching errors could lead to undesired consequences.

Fang, Bindu, and Gupta et al. [8] show that LLM agents, particularly GPT-4, can autonomously hack websites, albeit open source models less so. The study’s findings raise questions about widespread LLM’s deployments.

A recent example of Air Canada’s chatbot debacle offering inaccurate response to refund policies suggests oversight and human inspection are imperative [9].

Just as any new feature with immense possibilities of extending LLM’s scope takes time to adopt and use it properly, so does autonomous function calling. Such feature capabilities in both closed and open-source LLMs are still evolving to address their benefits securely and mitigate risks eventually. As the AI use cases in industry grow, and its proposed benefits in productivity, impact in society, and promise of economic growth, so does its demand for safety, security, and accuracy.

Note: The initial versions of function calling did not automatically call your functions; instead, it generated structured JSON arguments for you to extract and use to call your function. The later versions in GPT-3.5-turbo, GPT-4 and beyond do with parallel function calling and more methodically via the Assistant API tools, which I will cover in a separate blog.

To exercise some control over whether an LLM should invoke your function automatically, you can supply an additional parameter tool_choice to the Chat Completion API. (By default this value is None, indicating do call automatically.)

Limitations and drawbacks aside, function calling affords immediate and tangible practical use cases.

Examples and Use Cases of Function Calling in LLM

Apart from the above use cases mentioned in the Open AI programming guide [10], Ben Lorica visually and comprehensively captures use cases of general function calling in LLMs, including the OpenAI Assistant Tools API [11]. Lorica succinctly states that early use cases include applications such as customer service chatbots, data analysis assistants, and code generation tools. Other examples extend to creative, logistical, and operational domains: writing assistants, scheduling agents, summarizing news., etc.

Far more importantly, though, is its abiding ability to interact with external tools and ecosystems, to immense benefits.
